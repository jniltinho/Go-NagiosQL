## ADDED Requirements

### Requirement: Per-entity config file generation
The system SHALL generate Nagios `.cfg` files from the database using Go's `text/template`. Each host SHALL produce its own `hosts/{hostname}.cfg`. Each service group (by `config_name`) SHALL produce its own `services/{config_name}.cfg`. All other object types SHALL produce one file per type (e.g., `commands.cfg`, `timeperiods.cfg`).

#### Scenario: Host config file generated
- **WHEN** config generation is triggered for a specific host
- **THEN** `/usr/local/nagios/etc/nagiosql/hosts/{hostname}.cfg` is written with a valid `define host { }` block

#### Scenario: Service config file generated by group
- **WHEN** config generation is triggered for all services
- **THEN** one file per unique `config_name` is written under `/usr/local/nagios/etc/nagiosql/services/`

#### Scenario: Template objects excluded from register
- **WHEN** a host template with `register='0'` is generated
- **THEN** the generated block contains `register 0` and no `host_name` directive

### Requirement: Stale config detection
The system SHALL detect when a generated config file is older than the corresponding database record using `tbl_tablestatus.last_modified` versus filesystem `mtime`. Stale objects SHALL be visually indicated in the UI.

#### Scenario: Stale detection
- **WHEN** `tbl_tablestatus.last_modified` for `tbl_host` is newer than the mtime of `hosts/{hostname}.cfg`
- **THEN** the host is marked as "config outdated" in the `/admin/verify` view

### Requirement: Config targets — local, FTP, SSH
The config generation engine SHALL support three deployment methods as defined in `tbl_configtarget.method`: 1=local filesystem write, 2=FTP upload (optional FTPS), 3=SSH/SFTP upload via key or password auth.

#### Scenario: Local write
- **WHEN** the config target uses method=1
- **THEN** files are written directly to the local filesystem paths from `tbl_configtarget`

#### Scenario: FTP upload
- **WHEN** the config target uses method=2
- **THEN** generated files are uploaded to the configured FTP server and path using `github.com/jlaffaye/ftp`

#### Scenario: SSH/SFTP upload
- **WHEN** the config target uses method=3 with key-based auth
- **THEN** files are transferred via SFTP using `golang.org/x/crypto/ssh` with the configured key path

### Requirement: Nagios 3 vs 4 template variants
The config generation engine SHALL select template variants based on the `version` field in `tbl_configtarget`. Nagios 4-specific directives (e.g., `hourly_value`, `importance`) SHALL only appear in output when version=4.

#### Scenario: Version 4 template selected
- **WHEN** a config target has `version=4`
- **THEN** generated host template files include Nagios 4-only directives

#### Scenario: Version 3 template selected
- **WHEN** a config target has `version=3`
- **THEN** generated files only include Nagios 3-compatible directives

### Requirement: Long-line continuation
The config generation engine SHALL wrap directive values longer than 800 characters using Nagios line continuation syntax (backslash at end of line), matching original NagiosQL behavior.

#### Scenario: Long value wrapped
- **WHEN** a check_command argument string exceeds 800 characters
- **THEN** the generated `.cfg` line uses `\` continuation and splits across multiple lines

### Requirement: Backup of previous config files
Before overwriting a config file, the system SHALL copy the previous version to the backup directory configured in `tbl_configtarget` (`hostbackup`, `servicebackup`).

#### Scenario: Backup created on overwrite
- **WHEN** a host config file already exists and a new version is generated
- **THEN** the previous file is copied to the backup directory with a timestamp suffix before being overwritten
